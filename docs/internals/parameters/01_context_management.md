# Context Management Internal Parameters

本文记录 Prompt 上下文构建系统里当前仍然重要的内部参数。

这些参数并不等价于公开配置项。
其中一部分已经可以通过实例配置影响运行时行为，另一部分仍然只应由开发者在实现层维护。

## 适用范围

- `shinbot.agent.context`
- `shinbot.agent.workflow.conversation`
- `shinbot.core.bot_config_admin`

## 上下文块打包参数

来源：`shinbot/agent/context/context_stage_builder.py`

| 参数 | 默认值 | 作用 | 当前暴露状态 |
| --- | --- | --- | --- |
| `min_tokens` | `300` | 控制上下文 message 块的最短目标长度。在未达到该值前，不因为时间间隔而切块。 | 未暴露 |
| `max_tokens` | `1500` | 控制单个上下文 message 块的最大目标长度。若加入下一批连续发言会超出上限，则回退这批发言并另开新块。 | 未暴露 |
| `timeout_ms` | `10 * 60 * 1000` | 在已达到 `min_tokens` 的前提下，若下一批消息与当前块尾部间隔超过该值，则允许切块。 | 未暴露 |
| `gap_marker_ms` | `3 * 60 * 1000` | 同一上下文块内部，两条消息间隔超过该值时，会插入一条绝对时间标记。 | 未暴露 |

补充说明：

- 当前实现会尽量保证“同一发送者的连续发言”不被中途截断。
- 当映射表没有发生可见变化时，sealed 前缀块会直接复用，只重建尾部活块和新增消息。

## 别名重建参数

来源：`shinbot/agent/context/alias_table.py`

| 参数 | 默认值 | 作用 | 当前暴露状态 |
| --- | --- | --- | --- |
| `ALIAS_REBUILD_IDLE_MS` | `10 * 60 * 1000` | 会话静默多久后允许尝试重建映射表。 | 未暴露 |
| `ALIAS_ACTIVE_WINDOW_MS` | `24 * 60 * 60 * 1000` | 用于区分活跃成员和不活跃成员。24h 内活跃但未进入 A 组的成员，会进入 `P` 组。 | 未暴露 |
| `ALIAS_FREQUENT_LIMIT` | `10` | 发言最频繁的前 N 个成员使用 `A0` 到 `A9`。 | 未暴露 |

当前重建规则：

- 达到重建时机，不等于一定刷新上下文。
- 只有映射表发生“可见语义变化”时，才会触发上下文整体刷新。
- 如果只是 `A` 组或 `P` 组内部排序变化，而组成员本身没有变化，系统会尽量保留原 alias 编号，并把这次重建视为“无变化”。

## 上下文淘汰与压缩参数

来源：`shinbot/agent/context/eviction.py`、`shinbot/agent/workflow/conversation.py`

| 参数 | 默认值 | 作用 | 当前暴露状态 |
| --- | --- | --- | --- |
| `max_context_tokens` | `32000` | 请求总 token 达到该阈值时，触发上下文块淘汰检查。 | 已通过实例配置暴露 |
| `context_evict_ratio` | `0.6` | 触发淘汰后，按比例选择一批最旧的上下文块移出活动上下文。 | 已通过实例配置暴露 |
| `context_compression_max_chars` | `240` | 压缩转述模型输出的目标最大长度。 | 已通过实例配置暴露 |
| `max_compressed_entries` | `3` | 压缩记忆区最多保留多少条最新压缩结果。 | 未暴露 |

补充说明：

- 当前淘汰单位是 message 块，不是单条消息。
- 当前淘汰比例按块数量而不是块 token 权重计算，这是有意保留的实现取舍。

## 图片与消息编号相关参数

来源：`shinbot/agent/context/state_store.py`

| 参数 | 默认值 | 作用 | 当前暴露状态 |
| --- | --- | --- | --- |
| `message_ids.capacity` | `9999` | 上下文和指令 stage 中消息短编号的环形容量。 | 未暴露 |
| `image_ids.capacity` | `9999` | 图片短编号的环形容量。 | 未暴露 |

补充说明：

- 这两个容量的目的不是全局唯一，而是在单会话内提供足够长、足够稳定的短 ID。
- 如果未来消息规模或上下文复用策略变化，这两个容量可能一起调整。

## 维护建议

- 只有当参数会显著影响行为、token 开销、性能或缓存命中时，才值得在这里记录。
- 如果后续为这些参数设计前端或实例配置入口，应优先保持默认值与这里的说明一致。
- 如果代码行为已经改变，应先更新实现，再同步更新本文，不要让本文先于代码承诺新的行为。
