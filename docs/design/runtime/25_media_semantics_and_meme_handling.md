# ShinBot 技术规范：媒体语义与 Meme 处理 (Media Semantics & Meme Handling)

本文档定义 ShinBot 在会话工作流中的图片、表情包与其它媒体资源的语义处理策略。

目标不是把所有 `img` 一律交给多模态模型，也不是把所有媒体都先压扁成一句文本，而是建立一条可治理、可缓存、可回看、成本受控的媒体处理链路：

- 规范协议中的表情 / meme 优先按显式类型处理
- 普通 `img` 先进入统一 fingerprint 基础设施
- 常规图片默认由多模态模型理解
- meme 不默认反复交给多模态模型，而是优先复用已验证的短文本 digest
- 对历史 meme 与图片，系统必须支持“回看原图并重新按图片处理”
- 原始媒体文件、会话内出现记录与全局语义记录的生命周期分别治理

本文档是后续实现媒体归一化、meme 聚类、视觉缓存与上下文升级的主要设计依据。

---

## 1. 设计目标

### 1.1 区分普通图片与表情包

- 普通图片与 meme 在语义密度、重复频率、处理成本上差异显著。
- 设计必须支持：
  - `generic_image`
  - `meme_image`
  - `emoji_native`

### 1.2 避免重复触发多模态模型

- 表情包在单个会话中高度重复。
- 已确认语义的同图或近似图，不应在每次出现时都重新触发多模态识别。
- 重复检测必须先于昂贵模型调用发生。

### 1.3 保留原图回看能力

- 虽然 meme 日常可用短文本 digest 表示，但系统不能失去回看原图的能力。
- 当用户追问“图里是谁”“图中文字是什么”“这张表情包是哪位人物”时，LLM 必须能通过 Tool 重新请求图像级分析。

### 1.4 与现有上下文系统衔接

- 当前上下文主链路仍以文本 history 为主。
- 媒体语义层应为该链路提供：
  - 可进入上下文的文本 digest
  - 可回溯原图的资源句柄
  - 可供 attention 使用的重复性信号

---

## 2. 非目标

- 本文档不定义通用图像理解模型的完整路由管理。
- 本文档不要求首版支持复杂的图像标签体系或美学评分。
- 本文档不要求对所有图片首次出现都进行 meme 分类。
- 本文档不以“图像分类准确率最大化”为首要目标，而以运行成本和会话理解收益平衡为首要目标。

---

## 3. 媒体类型分层

### 3.1 `emoji_native`

表示协议本身对表情进行了显式封装的元素，例如：

- `emoji`
- `qq:mface`
- 未来其他平台的专有表情段类型

这类消息的特点：

- 上层协议已经明确提供“表情”语义
- 不必先当作普通图片处理
- 可直接进入表情语义通道

### 3.2 `meme_image`

表示实际以图片承载、但在语义上主要作为表情/梗图使用的媒体。

常见特征：

- 同一会话内高重复
- 常以单张图回复
- 文本语义强依赖场景、人物表情、梗图模板

### 3.3 `generic_image`

表示普通图片、截图、照片、素材图、图文内容图等。

这类图片通常：

- 首次出现的信息密度较高
- 值得优先由多模态模型直接理解
- 不应默认套用 meme 的重复与摘要策略

---

## 4. 总体处理原则

### 4.1 普通图片：Multimodal-First

对 `generic_image`：

- 首次出现时，默认应优先交给多模态模型理解
- 首轮解析后可沉淀结构化或文本化摘要
- 历史回放时可优先使用摘要，必要时再回看原图

### 4.2 Meme：Digest-First But Reanalyzable

对 `meme_image`：

- 不应默认每次都重新触发多模态模型
- 应优先复用已确认的 `meme_digest`
- digest 默认不超过 50 字，避免丢失关键语义又不污染上下文
- 必须支持通过 Tool 回看原图并按图片重分析

### 4.3 Hash 是全局基础设施，不是 Meme 特判功能

所有 `img` 都应统一进入媒体 fingerprint 流程：

- `raw_hash`
- `strict_dhash`

Hash 不应只在 `force_meme` 场景启用，因为正常用户也可能重复发送同一张表情或图片。

`force_meme` 的作用应仅是：

- 影响分类优先级
- 影响是否降低 LLM 检定阈值
- 不影响是否执行 hash / 缓存 / 去重

---

## 5. 媒体 Fingerprint

### 5.1 `raw_hash`

- 对原始媒体字节计算稳定哈希，建议使用 `sha256`
- 用于判断“完全同一文件”
- 也是跨会话复用语义记录的主键候选

### 5.2 `strict_dhash`

- 对图像计算严格视觉哈希
- 用于辅助识别“同图不同转码”“缩放后仍相同的表情包”
- 与 `raw_hash` 不同，`strict_dhash` 不应用作单独主判据

### 5.3 推荐阈值

当前建议：

- `raw_hash` 完全相同：视为同文件
- `strict_dhash` 距离 `< 5`：视为高度近似

说明：

- 该阈值刻意收紧，不采用宽松视觉去重
- 目的是降低把相似截图、相似海报、相似头像误判为同一 meme 的风险

---

## 6. Meme 检定主路径

### 6.1 不默认对所有图片做 Meme 分类

普通 `img` 首次出现时，不建议直接调用模型判断“是不是 meme”。

原因：

- 成本高
- 收益低
- 大量图片并不需要这一步

### 6.2 会话内重复触发检定

推荐主规则：

- 仅在同一 `session_id` 内
- 统计窗口为最近 `14 天`
- 当同一 `raw_hash` 重复出现次数 `>= 3`
- 或在严格条件下命中同一高相似 cluster

才触发一次 LLM 检定：

- “这是否属于 meme / 表情类图像”
- 若是，生成 `meme_digest`
- 若不是，标记为 `generic_image`

该策略的设计目标是：

- 利用“会话内高重复”这个高价值特征识别 meme
- 避免把一次性转发图片误判为表情包
- 把模型调用推迟到值得付费的时候

### 6.3 结果缓存

一旦某个 `raw_hash` 被模型确认：

- `kind = meme_image` 或 `generic_image`
- `meme_digest` 或图像摘要被持久化

后续再次命中时：

- 不再重复触发同类检定
- 优先复用历史语义结果

---

## 7. 协议优先分流

### 7.1 显式表情优先

若适配器已经把消息解析为显式表情元素：

- `emoji`
- `qq:mface`
- 其它专有表情元素

则应直接进入表情语义通道，而不是先按普通图片处理。

### 7.2 不规范 Bot 的兜底

部分 Bot 或插件会把表情包直接作为普通 `img` 发送。

对此系统必须保留：

- 图片级 fingerprint
- 会话内重复检测
- 发送者级媒体策略

以兜底处理这些“隐式 meme”。

---

## 8. 发送者媒体策略

### 8.1 目的

为已知行为模式稳定的 sender 提供分类偏置，而不是硬编码特判。

### 8.2 推荐策略值

- `auto`
- `prefer_meme`
- `force_meme`
- `force_generic_image`
- `ignore_media_semantics`

### 8.3 作用边界

发送者媒体策略只能影响：

- 分类优先级
- 是否降低 meme 检定阈值
- 是否优先走 meme digest 通道

不得影响：

- 是否执行 hash
- 是否缓存资源
- 是否允许原图回看

### 8.4 适用场景

典型例子：

- 某些表情管理 Bot 经常把表情包作为普通图片发出
- 可为其配置 `force_meme` 或 `prefer_meme`

这样可降低首次识别成本，但系统仍应允许在低置信度时回退到 `generic_image`。

---

## 9. Meme Digest

### 9.1 长度要求

`meme_digest` 应控制在 50 字以内。

目标：

- 足够承载基本语义
- 不在长上下文中造成冗余污染

### 9.2 内容要求

推荐优先覆盖以下信息中的 2 到 4 项：

- 情绪 / 态度
- 动作或表情
- 关键人物 / 对象
- 明显图中文字
- 语用作用，如附和、嘲讽、无语、催促

### 9.3 复用规则

对同一 `meme_cluster`：

- 不应每次重新生成长文本 digest
- 首次确认后应复用已有 digest
- 后续上下文可附带重复次数或最近出现信息

例如：

- `[表情: 无语摊手，像在吐槽]`
- `[表情: 熊猫头坏笑，带阴阳怪气语气]`
- `[重复表情: cluster_17，近两周第 4 次出现]`

---

## 10. 原图回看能力

### 10.1 必要性

虽然 meme 日常以 digest 进入上下文即可，但系统必须支持原图回看。

典型问题：

- “这张图里是谁？”
- “图里写了什么字？”
- “这是哪个角色？”
- “这张表情包里的人物原型是谁？”

### 10.2 推荐 Tool

首版建议至少支持：

- `media.inspect_original`
- `media.reanalyze_as_image`
- `media.ocr_original`

输入优先使用：

- `message_id`
- `asset_id`
- `raw_hash`
- `meme_cluster_id`

而不是要求模型直接拿外部 URL。

### 10.3 回看后的行为

回看原图后：

- 可以返回更细粒度的视觉分析结果
- 但不应自动覆盖已有 `meme_digest`
- 除非该次分析明确更新了更高质量的长期语义记录

---

## 11. 与上下文系统的关系

### 11.1 当前上下文主链路

当前上下文系统仍以文本 history 为主。

因此媒体语义层应输出：

- 面向 history 的短文本 digest
- 面向 Tool 的原图资源句柄

### 11.2 媒体进入上下文的推荐方式

- `emoji_native`：
  - 直接转成短表情语义
- `meme_image`：
  - 默认进入 `meme_digest`
- `generic_image`：
  - 首次优先经多模态理解
  - 历史阶段优先使用摘要

### 11.3 不推荐的方式

以下方式不应默认采用：

- 每轮都把历史图片重新发给多模态模型
- 纯图片消息完全没有任何文本语义占位
- 对重复 meme 持续执行 OCR 或视觉理解

---

## 12. Attention 与工作流集成

### 12.1 Attention 增量

媒体语义应影响 attention，但不应简单等于“有图就高权重”。

建议：

- `generic_image` 首次出现：中高 attention 增量
- 已确认 `meme_image` 首次出现：中等或偏低增量
- 重复 `meme_image`：低增量
- 高频重复 meme：可接近零增量，仅保留群体情绪信号

### 12.2 重复表情的收益

对高重复 meme：

- 不再反复请求模型理解
- 可直接作为“重复表情 cluster 事件”进入 workflow
- 仅保留数量、近期频率与基础语义

### 12.3 用户文本与 Meme 共存

若消息同时带有文本和图片：

- 文本仍应正常进入上下文
- meme 只作为附加语义，不应覆盖文本主体

---

## 13. 持久化建议

### 13.1 `media_assets`

用于保存原始媒体资源索引。

建议字段：

- `asset_id`
- `raw_hash`
- `mime_type`
- `storage_path`
- `byte_size`
- `width`
- `height`
- `last_seen_at`
- `expire_at`

### 13.2 `session_media_occurrences`

用于保存会话内重复统计。

建议字段：

- `session_id`
- `raw_hash`
- `strict_dhash`
- `first_seen_at`
- `last_seen_at`
- `occurrence_count`
- `expire_at`

说明：

- 该表主要服务于“会话内近两周重复 >= 3”的 meme 检定规则

### 13.3 `media_semantics`

用于保存全局媒体语义结果。

建议字段：

- `raw_hash`
- `strict_dhash`
- `kind`
- `digest`
- `verified_by_model`
- `meme_cluster_id`
- `first_seen_at`
- `last_seen_at`
- `expire_at`

---

## 14. Retention 与 Sliding TTL

### 14.1 原始图片文件

建议：

- `30 天 sliding TTL`

规则：

- 每次再次出现时，刷新 `last_seen_at`
- 同时重置 `expire_at = now + 30 days`
- 到期后允许删除原始文件

### 14.2 会话内出现记录

建议：

- `60 天 sliding TTL`

规则：

- 每次同会话再次出现时刷新 TTL
- 该层比 14 天检测窗口更长，便于重复统计与短期回放

### 14.3 全局语义记录

建议：

- `180 天 sliding TTL`
- 或“长期保留 + 容量淘汰”

理由：

- 语义记录体积小
- 可显著降低后续重复分析成本
- 即使原图过期删除，仍可复用语义结果

### 14.4 Sliding TTL 原则

本设计采用 sliding TTL，而不是固定过期时间。

也就是说：

- 资源再次出现时，应刷新 `last_seen_at`
- 同时重置 `expire_at`

这样：

- 常用 meme 会自然常驻
- 冷门资源会自然淘汰
- 不会因为短暂沉寂而丢失高价值语义缓存

---

## 15. 清理任务

### 15.1 清理顺序

建议定期任务按以下顺序清理：

1. 删除过期原始媒体文件
2. 删除过期会话内出现记录
3. 删除过期全局语义记录
4. 清理孤儿索引与不存在文件的资产记录

### 15.2 删除原则

删除原始文件后：

- 不应强制删除 `media_semantics`
- 只要语义记录仍在有效期内，就允许继续复用 digest

但此时：

- 若用户追问旧图细节，而原图已过期删除，则无法对“旧缓存原图”回看
- 只有在当前消息重新携带该图时，系统才可再次分析

---

## 16. 最小可行落地顺序

建议按以下顺序实施：

1. 为所有 `img` 引入 `raw_hash + strict_dhash`
2. 增加 `session_media_occurrences`
3. 实现“14 天窗口内重复 >= 3 才触发 meme 检定”
4. 增加 `media_semantics` 与 `meme_digest`
5. 为 `emoji_native` / `qq:mface` 接入显式表情语义通道
6. 增加 `media.inspect_original` 等回看 Tool
7. 增加原始文件与语义记录的 sliding TTL 清理任务

首版即使暂时不做复杂图像分类，也至少应保证：

- hash 是全局基础设施
- 显式表情优先分流
- 会话内重复触发 meme 检定
- meme digest 可复用
- 原图可回看
