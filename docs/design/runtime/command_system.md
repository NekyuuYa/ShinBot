# ShinBot 技术规范：指令系统 (Command System)

指令系统负责处理具有高度确定性和实时性要求的用户请求。指令在消息工作流中拥有最高优先级。

## 1. 灵活注册策略 (Loose Registration)

为了降低开发门槛并支持动态扩展，ShinBot 提供“双轨制”注册方案：

### 1.1 代理模式 (Delegated Mode)
插件开发者仅声明指令触发词，不定义参数结构。
- **行为**: 系统匹配到触发词后，将指令后的 **原始文本 (Raw Remaining Text)** 传递给插件。
- **优点**: 极简开发，插件拥有 100% 的解析自由度。

### 1.2 托管模式 (Managed Mode)
插件开发者通过 Pydantic 模型或类型标注声明详细的参数 Schema。
- **行为**: 系统自动解析字符串为结构化数据，并进行类型校验。
- **优点**: 框架代劳解析逻辑，支持自动生成帮助文档，支持参数缺省后的 **交互式补全 (Interactive Prompts)**。

## 2. 指令识别算法 (Resolution Logic)

识别遵循以下严格优先级：

### 2.1 优先级 P0：强制标记符 (Session Markers)
- **触发**: 消息头部匹配当前 `Session.config.prefixes` 中的标记符。
- **行为**: 提取首个单词匹配注册表。若未命中，**直接报错**（视为无效指令，不进入 Agent）。

### 2.2 优先级 P1：全量匹配 (Exact Match)
- **触发**: 消息全量等于指令名或别名（无需前缀）。

### 2.3 优先级 P2：正则匹配 (Regex Match)
- **触发**: 消息内容符合插件注册的正则表达式。

## 3. 交互式参数补全 (Interactive Filling)
对于“托管模式”，若缺少必填参数，框架将当前会话设为 `WaitingForInput` 状态，主动询问用户并捕获下一条消息作为输入，直到参数完整后再触发执行。
