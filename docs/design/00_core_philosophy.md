# ShinBot 核心思想：智能体原生与工作流架构

本文档阐述 ShinBot 区别于传统机器人框架的底层设计逻辑。

## 1. 核心定位：智能体原生 (Agentic-Native)

与传统“服务型”框架（侧重请求响应）不同，ShinBot 是为“决策型”智能体设计的。
- **语义完整性**: 所有的消息传输均保留 AST 结构，确保 Agent 能精准理解图片、引用、艾特等复杂上下文。
- **决策闭环**: 所有的 API 操作 (`call_api`) 均可作为智能体的 Tool 调用，实现逻辑与行为的自洽。

## 2. 三位一体链路 (Pillars)

我们将复杂的平台交互收敛为三条标准链路，实现核心调度与外部环境的解耦。

### 2.1 事件监听 (Listen / Ingress)
- 适配器负责“捕获并翻译”，将一切信号转换为 `UnifiedEvent` 注入工作流。

### 2.2 消息发送 (Speak / Egress)
- 核心产生平台无关的 **MessageElement 流**，由适配器在出口处执行“本地化渲染”。

### 2.3 API 请求 (Control / Action)
- 一个万能的 RPC 接口，涵盖了踢人、点赞等非消息流操作，支持 `internal` 命名空间透传。

## 3. 依赖解析与特化封装 (Dependency & Wrapper)

为了解决平台差异并复用逻辑，ShinBot 引入 **“依赖解析”** 机制：
- **分层封装**: 开发者可以编写“适配器增强插件”，专门封装特定平台的独有事件（如 QQ 的红包、Discord 的 Thread）。
- **解耦价值**: 业务插件只需声明其所需的依赖（如 `QQEventSupport`），即可获得针对该平台的增强处理能力，而核心引擎依然保持对具体协议的“零感知”。

## 4. 后续演进：语义化增强

ShinBot 致力于在底层链路之上提供更语义化的快捷方式（如 `ctx.reply()`, `ctx.poke()`），并最终通过 **兼容层 (Legacy ACL)** 实现对 AstrBot-Lite 等存量生态的平滑吸纳。
