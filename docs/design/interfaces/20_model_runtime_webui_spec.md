# ShinBot 技术规范：模型管理页 (Model Runtime WebUI Spec)

本文档定义 ShinBot WebUI 中“模型管理”页面的结构、交互和信息架构。

本页服务于 `18_agent_model_runtime.md` 与 `19_database_persistence_architecture.md`，目标是让 Provider、Route 和模型成员配置在一个页面内完成，而不是拆成多个割裂页面。

---

## 1. 页面定位

模型管理页是一个单独的业务页面，但它内部承载多个模型管理视角。

要求：

- 不单独提供“模型列表页”。
- Provider、Route、已配置模型成员，都在同一个页面中管理。
- 页面通过标签页切换不同能力域，而不是为“对话模型”“嵌入模型”“其它模型”分别做独立页面。

建议路由：

- `/model-runtime`

建议导航名称：

- `模型`
- 或 `模型管理`

---

## 2. 顶层信息架构

页面采用“两栏 + 标签页”结构。

### 2.1 顶部标签页

页面顶部固定为四个标签页：

- `路由`
- `对话`
- `嵌入`
- `其他`

含义：

- `路由`
  - 管理逻辑路由组、成员模型、优先级与策略。
- `对话`
  - 管理可用于聊天/生成的 Provider 与其模型。
- `嵌入`
  - 管理 embedding Provider 与其模型。
- `其他`
  - 管理暂不归入前两类的能力，例如 moderation、rerank、image、audio 等。

标签页切换的作用是切换“当前管理域”，而不是切换不同页面。

---

## 3. 页面布局

### 3.1 左侧面板

左侧面板宽度约为页面的 `1/3`。

职责：

- 展示当前域下的 `Route 列表`
- 展示当前域下的 `Provider 列表`
- 作为当前右侧详情区的选择入口

左侧面板不使用表格，使用两个分组卡片：

1. `路由列表卡片`
2. `供应商列表卡片`

两张卡片可上下堆叠。

约束：

- 同一时刻只允许一个对象处于“当前编辑中”状态。
- 当前选中对象必须有明显高亮。
- 支持搜索，但不优先做复杂筛选器。

### 3.2 右侧面板

右侧面板宽度约为页面的 `2/3`。

右侧面板不做成“长表单大杂烩”，而是拆成卡片编组。

核心要求：

- 设置区独立成卡片
- 高级设置独立成卡片
- 已配置模型独立成卡片

也就是说，右侧主内容由三张核心卡片构成：

1. `基础设置卡片`
2. `高级设置卡片`
3. `已配置模型卡片`

---

## 4. 视图切换规则

左侧可以选中两类对象：

- `Route`
- `Provider`

右侧内容根据当前选中对象切换，但仍保持统一三卡片骨架。

### 4.1 选中 Provider 时

右侧主要编辑的是 Provider 及其下属模型。

三张卡片含义为：

1. `基础设置卡片`
   - Provider 基础信息
2. `高级设置卡片`
   - 请求头、代理、可选高级能力
3. `已配置模型卡片`
   - 当前 Provider 下的模型成员

### 4.2 选中 Route 时

右侧主要编辑的是 Route 及其成员模型。

三张卡片仍保留，但语义变为：

1. `基础设置卡片`
   - Route ID、用途、策略、启用状态
2. `高级设置卡片`
   - fallback / sticky / 条件策略等高级项
3. `已配置模型卡片`
   - 当前 Route 的成员模型与优先级

这样可以避免“选了 Route 就跳另一套布局”，保持整个页面稳定。

---

## 5. 标签页与左侧列表的关系

标签页会过滤左侧两组列表的内容。

### 5.1 `路由` 标签

左侧显示：

- 以 Route 列表为主
- Provider 列表作为辅助引用

适合做：

- 路由策略管理
- 路由成员编组
- fallback / weighted / priority 配置

### 5.2 `对话` / `嵌入` / `其他` 标签

左侧显示：

- 以 Provider 列表为主
- Route 列表作为辅助入口

适合做：

- 新建 Provider
- 配置 URL / Token / Header / Proxy
- 从供应商拉取模型列表
- 将模型加入某个 Route

---

## 6. 基础设置卡片

基础设置卡片是右侧第一张卡片。

### 6.1 Provider 模式下

至少包含：

- `id`
- `显示名称`
- `url`
- `token`
- `供应商来源`

其中“供应商来源”用于快速填充基础 URL、默认请求头或已知能力模板。

例如：

- OpenAI
- Azure OpenAI
- OpenRouter
- Ollama
- Gemini
- Anthropic
- 自定义 OpenAI 兼容源

行为要求：

- 选择供应商来源后，可自动填充 `url`
- 自动填充必须是可编辑的，而不是只读模板
- `token` 默认输入态必须遮罩
- API 返回中不回显敏感 token

### 6.2 Route 模式下

基础设置卡片改为 Route 基础项：

- `id`
- `显示名称` 或 `用途`
- `策略`
- `启用状态`
- `绑定域`
  - 对话 / 嵌入 / 其他

---

## 7. 高级设置卡片

高级设置卡片是右侧第二张卡片。

### 7.1 Provider 模式下

至少包含：

- `请求头`
- `代理地址`
- `思考配置`
- `过滤配置`

设计要求：

- `请求头` 使用 key-value 编辑器，而不是纯 JSON textarea
- `代理地址` 支持空值
- `思考配置` 只有当前供应商/模型能力可用时才显示
- `过滤配置` 同理，只有可用时显示

“只在可用时出现”是硬性要求，不要展示一堆对当前源无效的灰色字段。

### 7.2 Route 模式下

高级设置卡片用于路由策略：

- `sticky session`
- `故障切换`
- `超时覆盖`
- `条件限制`
- `权重策略`

要求：

- 如果当前 Route 策略是 `priority`，则权重字段可以隐藏或折叠
- 如果当前 Route 策略是 `weighted`，则权重字段转为主显示
- 不要所有策略字段同时平铺

---

## 8. 已配置模型卡片

这是右侧第三张核心卡片，也是页面最重要的操作区。

职责：

- 展示当前 Provider 下已配置的模型
- 或展示当前 Route 下已编组的模型成员
- 承载“拉取模型列表并添加”的操作

### 8.1 Provider 模式下

卡片内应支持：

- `获取模型列表`
- `手动添加模型`
- `测试连接`
- `全局开关`
- `单模型单独配置`

展示方式建议：

- 卡片内部使用二级卡片列表
- 每个模型是一张小卡片

每张模型卡至少包含：

- 模型名
- LiteLLM 模型标识
- 能力标签
- 启用状态
- 是否已加入 Route
- 单独配置入口

### 8.2 Route 模式下

卡片内应支持：

- 从现有 Provider 模型中添加成员
- 调整优先级
- 调整权重
- 启停单个成员
- 移出 Route

每个成员模型建议仍以小卡片方式展示，而不是表格。

原因：

- 和 Provider 模式视觉一致
- 更适合在卡片内承载开关、测试、配置动作

---

## 9. “获取模型列表”交互

这是模型管理页的重要能力。

要求：

- 用户在 Provider 模式下可点击 `获取模型列表`
- 前端请求后端探测该 Provider 能返回的模型列表
- 返回结果先进入选择面板，不直接写入系统配置
- 用户确认后，再将选中的模型加入当前 Provider

也就是说：

- “拉取模型列表”
- “选择要纳入管理的模型”
- “写入数据库”

必须是三步，而不是自动全量导入。

原因：

- 不同 Provider 可能返回大量模型
- 很多模型不需要进入正式管理面
- 避免把 UI 变成噪音堆积区

---

## 10. 测试连接交互

测试连接是 Provider 级能力，不是模型页外链工具。

要求：

- 在 Provider 模式的基础设置卡片或已配置模型卡片中提供
- 至少支持：
  - Provider 连接测试
  - 指定模型测试

测试结果至少显示：

- 是否成功
- 响应耗时
- 失败原因
- 返回的目标模型名或供应商响应标识

不要只弹一个“连接成功”的 toast 就结束。

---

## 11. 没有独立模型列表页

这是本页设计的硬约束。

不应存在：

- `/models` 独立大列表页
- “先去模型页建模型，再回路由页绑定”的双跳流程

正确流程应是：

- 在 Provider 视图中管理模型
- 在 Route 视图中编组模型
- 在一个页面内通过标签页和左侧列表完成切换

这样用户的心智模型会更稳定：

- Provider 管理来源
- Route 管理用途
- 模型作为两者之间的成员对象存在

---

## 12. 组件拆分建议

页面建议拆成以下组件：

- `ModelRuntimeTabs`
- `ModelRuntimeSidebar`
- `RouteListCard`
- `ProviderListCard`
- `ProviderSettingsCard`
- `ProviderAdvancedCard`
- `ProviderModelsCard`
- `RouteSettingsCard`
- `RouteAdvancedCard`
- `RouteMembersCard`
- `ModelConfigCard`
- `ConnectionTestDialog`
- `FetchModelsDialog`

注意：

- 左侧是“列表卡片”
- 右侧是“设置卡片”
- 模型成员本身也用卡片，而不是内嵌复杂表格

---

## 13. 状态管理建议

Pinia 应按业务域拆分，而不是把整个页面塞进一个超大 store。

建议拆分：

- `modelRuntimeProvidersStore`
- `modelRuntimeRoutesStore`
- `modelRuntimeCatalogStore`
- `modelRuntimeExecutionsStore`

其中：

- `catalogStore`
  - 负责“获取模型列表”和临时选择结果
- `executionsStore`
  - 后续可接监控和测试连接结果

---

## 14. API 对齐建议

当前后端已具备：

- Provider CRUD
- Model CRUD
- Route CRUD
- Execution list

但为了贴合本页 UI，后续建议再补两类接口：

1. `Provider capability probe`
   - 用于测试连接和探测能力
2. `Provider model catalog fetch`
   - 用于从供应商拉取模型列表供用户选择导入

注意：

- 虽然后端仍保留独立的 model API
- 但前端不应把它表现成“独立模型页面”

---

## 15. 视觉与交互原则

- 不做企业后台式三层表格堆叠。
- 尽量使用卡片、分区标题和状态 chip 构建信息层次。
- 左侧强调导航与切换，右侧强调配置与操作。
- 每次只突出一个主对象，避免同时展开多个大型编辑器。
- “可选高级能力”必须按能力显隐，不做无条件堆砌。

---

## 16. 首版实现建议

首版页面建议优先实现：

1. 单页面路由与四个标签页
2. 左侧 Route / Provider 双列表卡
3. Provider 模式三张卡片
4. Route 模式三张卡片
5. 测试连接
6. 获取模型列表并导入

可以延后实现：

- 复杂批量操作
- 运行指标可视化图表
- 多条件高级筛选
- 拖拽排序

---

## 17. 最终原则

这页的核心不是“把所有模型字段露出来”，而是让用户清楚回答三个问题：

- 模型从哪里来
- 模型用于什么路由
- 当前是否可用、可测、可替换

因此页面必须优先强化：

- Provider 作为来源
- Route 作为用途
- 模型作为成员

而不是把模型本身做成一个脱离上下文的大表。

---

## 18. 参考实现映射

本页可参考外部项目中 `ProviderPage` 的布局组织方式，但只参考信息结构，不直接复用其视觉风格。

参考来源：

- `/home/Nekyuu/Workplace/AstrBot-Lite/dashboard/src/views/ProviderPage.vue`

可借鉴的点：

- 顶部标签页切换能力域
- 左侧独立列表面板
- 右侧“基础配置 / 高级配置 / 模型成员”分区
- Route 模式与 Provider 模式共用同一页面骨架
- 模型成员区使用卡片/列表项承载操作，而不是大表格

不应直接照搬的点：

- 更重的 dashboard shell 风格
- 与当前项目不一致的字体、密集度和阴影体系
- 过多的后台表单感分割线
- 偏复杂的内嵌动画和信息噪声

ShinBot 应保留当前既有风格：

- 顶栏和侧栏沿用现有 `MainLayout`
- 内容区继续使用当前浅底色 island 式承载
- 卡片圆角、边框、按钮、状态 chip 保持与 `Instances` / `Plugins` 页面一致
- 视觉重点放在“结构清晰”和“状态可读”，而不是视觉炫技

### 18.1 参考后的具体映射

将参考页的结构映射到 ShinBot 时，建议如下：

- 参考页的 `ProviderSourcesPanel`
  - 映射为 ShinBot 左侧的 `RouteListCard + ProviderListCard`
- 参考页的右侧 `provider-settings-panel`
  - 映射为 ShinBot 右侧三张独立卡片，而不是一个超大卡片里塞所有内容
- 参考页中 router 特殊模式
  - 映射为 ShinBot 的 `Route` 选中态
- 参考页中的“已配置模型管理”
  - 映射为 ShinBot 的 `已配置模型卡片`
- 参考页中的“测试连接”和“手动添加模型”
  - 在 ShinBot 中保留为卡片内主动作和对话框，不额外拆新页面

### 18.2 保持当前项目一致性的硬约束

- 左侧宽度仍维持 `1/3`，右侧 `2/3`
- 右侧始终是多卡片编组，不退化成单一大面板
- 不新增独立模型总表页
- Route 与 Provider 的切换不改变页面整体骨架
- 如果参考实现与当前项目风格冲突，以当前项目风格优先
